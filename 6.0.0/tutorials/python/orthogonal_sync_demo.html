Orthogonal Regions with Sync Demo (Python)

This tutorial extends the basic orthogonal regions pattern by adding a JoinState synchronization point. Synchronization allows independent regions to coordinate at specific points during execution. This is essential in robotics for behaviors like "wait for both arms to reach a waypoint before proceeding" or "synchronize sensor fusion data from multiple sources."

Background

The JoinState is a special state that acts as a rendezvous point across orthogonal regions. All regions that share a JoinState with the same identifier must reach it before any of them can proceed. Once all participating regions arrive at the join point, they are all released simultaneously to continue their individual workflows. This pattern is useful for:

  • Synchronizing phases of a multi-region mission (e.g., all subsystems ready)
  • Coordinating sequential operations across independent state machines
  • Implementing barrier patterns where regions must align before proceeding
  • Modeling parallel-then-merge workflows with shared dependencies

1. Create the WorkerState

A simple looping state that counts iterations and returns "working" or "done". This is the same state used in the basic orthogonal demo. 

class WorkerState(State):
    def __init__(self, name: str, max_count: int, sleep_ms: int = 300) -> None:
        super().__init__(["working", "done"])
        self.name = name
        self.counter = 0
        self.max_count = max_count
        self.sleep_ms = sleep_ms

    def execute(self, blackboard: Blackboard) -> str:
        yasmin.YASMIN_LOG_INFO(
            f"WorkerState [{self.name}]: iteration {self.counter + 1}/{self.max_count}"
        )
        time.sleep(self.sleep_ms / 1000.0)

        self.counter += 1
        if self.counter >= self.max_count:
            return "done"
        return "working"

2. Create Region A with Synchronization

Region A runs 3 iterations of WORK_1, then transitions to a JoinState with identifier "mid_sync". The join state pauses execution until all other regions with the same identifier also reach their join point. When synchronized, Region A continues to WORK_2 for 2 more iterations before completing. 

def main() -> None:
    rclpy.init()
    set_ros_loggers()
    yasmin.YASMIN_LOG_INFO("yasmin_orthogonal_sync_demo")

    SYNC_ID = "mid_sync"

    # Region A: 3 iterations -> JoinState -> 2 more iterations
    reg_a = StateMachine(outcomes=["done"])
    reg_a.set_name("Region A")
    reg_a.add_state(
        "WORK_1",
        WorkerState("A.1", 3, 200),
        transitions={"working": "WORK_1", "done": "SYNC"},
    )
    reg_a.add_state(
        "SYNC",
        JoinState(SYNC_ID),
        transitions={"joined": "WORK_2"},
    )
    reg_a.add_state(
        "WORK_2",
        WorkerState("A.2", 2, 200),
        transitions={"working": "WORK_2", "done": "done"},
    )
    reg_a.set_start_state("WORK_1")

3. Create Region B with Synchronization

Region B runs 2 iterations of WORK_1 (faster than A), then reaches the same JoinState with "mid_sync". It waits at the join point until Region A also arrives. After synchronization, Region B continues to WORK_2 for 3 more iterations. This demonstrates how the faster region (B) waits at the barrier for the slower region (A). 

    # Region B: 2 iterations -> JoinState -> 3 more iterations
    reg_b = StateMachine(outcomes=["done"])
    reg_b.set_name("Region B")
    reg_b.add_state(
        "WORK_1",
        WorkerState("B.1", 2, 200),
        transitions={"working": "WORK_1", "done": "SYNC"},
    )
    reg_b.add_state(
        "SYNC",
        JoinState(SYNC_ID),
        transitions={"joined": "WORK_2"},
    )
    reg_b.add_state(
        "WORK_2",
        WorkerState("B.2", 3, 200),
        transitions={"working": "WORK_2", "done": "done"},
    )
    reg_b.set_start_state("WORK_1")

4. Create the OrthogonalState Container

The OrthogonalState is configured the same way as the basic demo. The outcome_map defines "success" when both regions complete. The JoinState synchronization is handled internally by the OrthogonalState container. 

    # Orthogonal state runs both regions in parallel with sync point
    ort = OrthogonalState(
        default_outcome="timeout",
        outcome_map={"success": {"A": "done", "B": "done"}},
    )
    ort.set_description("Runs two regions that sync at a JoinState before continuing.")
    ort.add_region("A", reg_a)
    ort.add_region("B", reg_b)

5. Create the Root State Machine

The root state machine wraps the OrthogonalState and handles its outcomes. This is identical to the basic orthogonal demo. 

    # Root state machine
    sm = StateMachine(outcomes=["success", "timeout"], handle_sigint=True)
    sm.set_description("Demonstrates orthogonal state with JoinState synchronization.")
    sm.add_state(
        "PARALLEL",
        ort,
        transitions={"success": "success", "timeout": "timeout"},
    )
    sm.set_start_state("PARALLEL")

    YasminViewerPub(sm, "YASMIN_ORTHOGONAL_SYNC_DEMO")

    try:
        outcome = sm()
        yasmin.YASMIN_LOG_INFO(outcome)
    except Exception as e:
        yasmin.YASMIN_LOG_WARN(e)

    if rclpy.ok():
        rclpy.shutdown()


if __name__ == "__main__":
    main()

6. Run the Demo

Execute the orthogonal synchronization demonstration to observe how JoinState coordinates region execution. 

ros2 run yasmin_demos orthogonal_sync_demo.py

How It Works

The JoinState synchronization follows this flow:

  • Region A starts WORK_1 (3 iterations, ~600ms total at 200ms each)
  • Region B starts WORK_1 (2 iterations, ~400ms total at 200ms each)
  • Region B reaches the JoinState first and waits for Region A
  • Region A finishes WORK_1 and reaches the same JoinState
  • Both regions are released simultaneously to their WORK_2 states
  • Region A runs WORK_2 (2 iterations), Region B runs WORK_2 (3 iterations)
  • Both complete independently
